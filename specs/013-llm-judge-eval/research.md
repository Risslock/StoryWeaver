# Research: LLM-as-Judge Response Evaluation

**Feature**: 013-llm-judge-eval | **Date**: 2026-06-28

---

## R-001: EvaluationRecord Storage Format

**Decision**: SQLite table (`data/eval.db`, model `ResponseEvalRecord`) via the existing `packages/storage/` SQLAlchemy 2.x + aiosqlite infrastructure.

**Rationale**: Three requirements make JSONL unworkable: (1) in-place score updates (scored records must have judge fields written back without rewriting the whole file), (2) `judge_status`-based filtering for skip logic (FR-014), and (3) run-scoped querying (FR-013). SQLite handles all three natively. Both the harness CLI and the Gradio UI share the same `data/eval.db` store, so results written via `eval_runner.py` are visible from the UI and vice versa.

**Alternatives considered**:
- JSONL (append-only) â€” rejected: no partial-record updates; would require full file rewrite to add judge scores to existing rows.
- ChromaDB collection â€” rejected: designed for vector search, not relational eval records; no natural primary key or status filtering.
- Separate PostgreSQL â€” rejected: violates Principle IV (local-first); adds infra dependency.

## R-002: Generation Step (Answer Producer)

**Decision**: Call the existing `ask_question(question, campaign_id, role)` function from `apps/web/services/knowledge.py` directly. No new generation component (`generator.py`, `AnswerGenerator` class, `ANSWER_PROVIDER`/`ANSWER_MODEL` env vars) is built.

**Rationale**: Spec clarification (2026-06-28) confirmed that the goal is to evaluate the production pipeline already used in the Knowledge Q&A Gradio tab. `ask_question()` performs the full pipeline: multi-query expansion â†’ `ChromaKnowledgeRetriever` â†’ RRF ranking â†’ `OllamaProvider` synthesis â†’ returns `(answer_text, list[KnowledgeChunk])`. Building a separate AnswerGenerator would create a divergence risk: the harness would evaluate a pipeline different from production, defeating the purpose.

**Callers**:
- CLI (`eval_runner.py`): receives `campaign_id` via required `--campaign-id` UUID arg and `role` via required `--role` arg
- UI (`response_eval.py` service â†’ `rag_eval.py` handler): reads `campaign_id` and `role` from the active `CampaignSession` in Gradio `session_state`

**Provider control**: The `ask_question()` pipeline uses `KNOWLEDGE_LLM_MODEL` (and its embedding/enrichment siblings). These env vars remain unchanged; no new generation-specific env vars are introduced.

**Alternatives considered**:
- Build new `AnswerGenerator` class â€” rejected: evaluates a different pipeline from the production Knowledge Q&A tab; divergence risk.
- Add `ANSWER_PROVIDER`/`ANSWER_MODEL` env vars â€” rejected: `ask_question()` already has its provider configured; adding parallel vars splits configuration and causes confusion about which model is under test.

## R-003: Structured Judge Output Parsing

**Decision**: Use `LLMProvider.generate_structured(prompt, response_type=JudgeScore, system=...)` where `JudgeScore` is a Pydantic model. The existing `generate_structured()` default implementation calls `generate()` + `model_validate_json()`, which is sufficient. No external structured-output library is needed.

**Rationale**: `LLMProvider.generate_structured()` already exists in `packages/llm/llm/interface.py` and is the established pattern. It parses the LLM's JSON response into a Pydantic model, and on parse failure raises a `ValidationError` that maps cleanly to `judge_status: "parse_error"`.

**Alternatives considered**:
- Regex extraction from natural language response â€” rejected: fragile, high parse-error rate, maintenance burden.
- Function/tool calling (Anthropic tool_use) â€” rejected: provider-specific API; `generate_structured()` abstraction already handles this at the provider level if needed.

## R-004: Run ID Generation

**Decision**: `run_id` is a string of the form `YYYYMMDD-HHMMSS-{8-char-uuid4-prefix}` generated at the start of `eval_runner.py`. Example: `20260628-143022-a3f8c1b9`. UI-triggered runs use the same format, generated by `response_eval.py` at the start of each "Run Response Quality Eval" invocation.

**Rationale**: Human-readable and sortable (timestamp prefix), unique (UUID suffix prevents collision from two runs started in the same second). Stored as a plain string in the SQLite `run_id` column (indexed for filtering). CLI and UI runs are indistinguishable at the storage level â€” both are `ResponseEvalRecord` rows with a `run_id`.

**Alternatives considered**:
- Pure UUID4 â€” rejected: opaque, not sortable, hard to recognize in logs.
- Sequential integer â€” rejected: not portable across machines; race conditions if two runs start simultaneously.

## R-005: Judge Prompt Template Location

**Decision**: Default template at `packages/rag/rag/evaluation/prompts/judge_prompt.txt`. Overridable via `JUDGE_PROMPT_PATH` env var (absolute or repo-relative path). If `JUDGE_PROMPT_PATH` is set and the file does not exist, `judge_runner.py` and `response_eval.py` exit/raise with a clear error.

**Rationale**: FR-007 requires the template be modifiable without code changes. Env-var override is the established pattern for configurable paths in this project. Shipping a default template inside the package makes it work out of the box without any env var.

**Alternatives considered**:
- Store prompt in a DB record â€” rejected: adds schema complexity; prompts are code-adjacent configuration, not user data.
- Hardcode prompt string in `judge.py` â€” rejected: explicitly prohibited by FR-007.

## R-006: Judge Provider Factory Pattern

**Decision**: New `get_judge_provider(provider: str, model: str) -> LLMProvider` function in `packages/rag/rag/evaluation/factory.py`. Reads `JUDGE_PROVIDER` and `JUDGE_MODEL` env vars. Raises `EnvironmentError` with a descriptive message listing valid providers if either is missing or unrecognised.

**Rationale**: Identical pattern to `get_knowledge_enrich_provider()` in `packages/rag/rag/knowledge/factory.py`. No `get_answer_provider()` is needed â€” generation uses `ask_question()` which manages its own provider internally.

**Supported providers**:
| `JUDGE_PROVIDER` | Class | Required additional env var |
|---|---|---|
| `ollama` | `OllamaProvider` | `OLLAMA_BASE_URL` (optional, defaults to `http://localhost:11434`) |
| `claude` | `AnthropicProvider` | `ANTHROPIC_API_KEY` |

## R-007: Context Truncation Strategy

**Decision**: A configurable maximum character budget for context passed to the judge, defaulting to 8,000 characters (roughly 2,000 tokens). Controlled by `JUDGE_MAX_CONTEXT_CHARS` env var. When context exceeds this budget, chunks are included greedily from highest RRF score downward until the budget is exhausted; remaining chunks are dropped. The record is annotated with `judge_context_truncated: true`.

**Rationale**: Most local Ollama models have a 4Kâ€“8K token context window. Passing the full top-10 chunks could easily overflow. 8,000 characters is a conservative safe default. Both CLI and UI paths use the same truncation logic inside `JudgeEvaluator`.

**Alternatives considered**:
- Token-count-based truncation â€” rejected: requires a tokenizer per provider; adds a transformers dependency to the eval path. Character-count approximation is sufficient for this use case.
- Always pass all chunks â€” rejected: would cause silent truncation or errors at the provider level for most local models.

## R-008: UI Integration Pattern for RAG Evaluation Tab

**Decision**: Extend the existing `apps/web/pages/gm/rag_eval.py` with a new "Response Quality" accordion/section below the existing retrieval results section. A new thin service `apps/web/services/response_eval.py` exposes an async generator `run_response_eval(questions, campaign_id, role)` that calls `ask_question()` then `JudgeEvaluator.evaluate()` per question and yields `(EvaluationRecord, JudgeResult)` tuples. The Gradio handler in `rag_eval.py` iterates this generator, streams per-question progress, and accumulates a results table.

**Session state integration**: `CampaignSession` (already in `session_state`) provides `campaign_id` (UUID) and `role` ("gm"|"player") â€” no additional UI inputs are needed for campaign scoping. The Response Quality section is enabled only when a campaign is active.

**Provider configuration**: `JUDGE_PROVIDER` and `JUDGE_MODEL` are read from env vars at startup. The UI does not expose judge provider selection â€” this is a developer/admin setting, not a per-session user setting.

**Placeholder-first**: On first render (before any judge run), the Response Quality section shows a stub: "Response Quality Evaluation â€” run retrieval eval first, then click Run Response Quality Eval". The Run Response Quality Eval button is enabled only when (a) a campaign session is active and (b) a JSONL question file has been selected.

**Error handling**: Any `ProviderUnavailableError` or `EnvironmentError` from the judge is caught in the Gradio handler and surfaced as a visible error message (never log-only per Principle VII).

**Alternatives considered**:
- New separate Gradio tab â€” rejected: user explicitly requested "integrated with the RAG Evaluation tab"; a new tab duplicates the JSONL input + campaign context and adds navigation friction.
- Inline scoring alongside retrieval run ("Run Eval" triggers both) â€” rejected: spec requires a separate judge step; LLM judge calls make retrieval eval unpredictably slow; users should be able to run retrieval only.
- Ephemeral in-session results (no persistence) â€” rejected: harness and UI sharing the same `data/eval.db` store means CLI-run records are visible in the UI and vice versa, enabling cross-tool comparison.
