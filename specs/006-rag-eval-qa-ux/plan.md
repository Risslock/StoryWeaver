# Implementation Plan: RAG Evaluation Tab & Q&A Source Visibility

**Branch**: `006-rag-eval-qa-ux` | **Date**: 2026-06-22 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `/specs/006-rag-eval-qa-ux/spec.md`

## Summary

Add a GM-only "RAG Evaluation" tab to the Gradio dashboard that loads a JSONL test file
and computes MRR, nDCG, and Recall@k for each question using the existing
`ChromaKnowledgeRetriever`. Metric logic lives in a new `evaluator.py` module inside
`packages/rag/`. A new `services/eval.py` bridges the UI to the evaluator. Separately,
the Q&A tab is updated to display the LLM answer and cited sources in distinct areas:
answer in the chatbot, sources in a collapsed `gr.Accordion` revealed on demand.

## Technical Context

**Language/Version**: Python 3.11+

**Primary Dependencies**: Gradio 4.x (UI), ChromaDB (vector store via existing RAG package),
Pydantic v2 (data models), `packages/rag` (retriever + evaluator), `packages/llm` (Ollama)

**Storage**: In-memory / ephemeral Gradio `gr.State` only — no new DB tables or migrations

**Testing**: `pytest` for unit (metric functions) and integration (eval service);
existing harness runner for retrieval regression scenarios

**Target Platform**: Local development server (same as all other Gradio tabs)

**Project Type**: Gradio web application (existing)

**Performance Goals**: Evaluation of a 100-question+ JSONL file completes without UI timeout;
each question's retrieval runs sequentially (acceptable for GM tooling, not user-facing latency)

**Constraints**: MUST reuse `ChromaKnowledgeRetriever.search()` — no separate retrieval path;
no new packages; no new DB tables; Gradio accordion (not custom JS) for sources toggle

**Scale/Scope**: Single GM per session; JSONL files expected to be small (10–500 questions);
no concurrent evaluation runs expected

## Phase 0: Research Summary

See [research.md](research.md) for full details. Key decisions:

- **Retriever**: `ChromaKnowledgeRetriever.search(role="gm", top_k=k)` — GMs see all chunks, no filter
- **Keyword matching**: Case-insensitive substring match on `KnowledgeChunk.text`
- **Q&A split**: Answer text → chatbot; citations → `gr.Accordion` (closed by default); `services/knowledge.py` signature unchanged
- **GM access**: Eval tab added to `gm_col` only — inherently GM-restricted by existing nav state machine
- **JSONL schema**: `{question, keywords, reference_answer, category}` — matches provided `test.py`

## Phase 1: Design Summary

See [data-model.md](data-model.md) and [contracts/evaluator_api.md](contracts/evaluator_api.md).

### Key design decisions

**`evaluator.py`** — pure functions, no async, no I/O:
- `calculate_mrr(keyword, chunks) -> float`
- `calculate_ndcg(keyword, chunks, k) -> float`
- `calculate_recall_at_k(keyword, chunks, k) -> float`
- `evaluate_question(test, chunks, k) -> RetrievalEvalResult`
- `aggregate_results(results) -> EvalSummary`

**`test_questions.py`** — Pydantic model + file loader:
- `TestQuestion(BaseModel)` with fields: `question`, `keywords`, `reference_answer`, `category`
- `load_test_questions(file_path) -> list[TestQuestion]` — raises `ValueError` with row+field on bad input

**`services/eval.py`** — async, calls retriever per question:
- `run_evaluation(file_path, campaign_id, k) -> tuple[list[RetrievalEvalResult], EvalSummary]`
- Logs at INFO (run start/end) and ERROR (per-question failures)

**`pages/gm/rag_eval.py`** — Gradio tab:
- File upload (`.jsonl`) + k slider (1–20, default 10)
- "Run Evaluation" button
- Placeholder markdown when no file loaded
- Results: `gr.Dataframe` (per-question table) + summary markdown
- Drill-down: selecting a row populates a `gr.Markdown` detail panel

**`pages/gm/knowledge_qa.py`** (modified):
- Remove citation suffix concatenation from `on_ask()` (lines 202–209)
- Add `sources_md = gr.Markdown("")` inside `gr.Accordion("Show sources", open=False)`
- `on_ask()` returns `(updated_history, "", formatted_citations)` — three outputs instead of two
- Wire new output to `sources_md`

**`app.py`** (modified):
- Import and call `build_rag_eval_page(session_state)` inside `gm_col` tabs
